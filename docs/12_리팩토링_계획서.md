## 1. Executive Summary

본 문서는 `UnitConverter.java`(루트 진입점 + 파사드) 및 `com.unitconverter.App`에 대한 **코드 스멜·ECB 혼재 분석**을 바탕으로, **동작·출력 계약을 유지한 채** 구조만 개선하는 Refactoring 계획을 정의한다.

| 원칙 | 내용 |
|------|------|
| 계약 불변 | TC-A/B, Golden Master, ERR-* 메시지 패턴 유지 |
| 테스트 선행 | `mvn test` + `golden_master` GREEN 후 Wave 진행 |
| 중복 제거 | 이미 존재하는 BCE 패키지(`entity` / `boundary` / `data`)로 위임 |
| OCP 유지 | 단위 추가 시 `UnitRegistry.register` 또는 config만 변경 |

## 2. 현황 요약 (분석 근거)

### 2.1 파일 구분

| 경로 | 역할 | ECB 혼재 |
|------|------|----------|
| 루트 `UnitConverter.java` | `main()` → `App.main` 위임만 | **없음** (양호) |
| `com.unitconverter.App.main()` | CLI 오케스트레이션 + `printf` 출력 | Control + Boundary 혼재 |
| `src/main/java/UnitConverter.java` | GREEN 파사드 (테스트 대상 API) | Entity + Control + Boundary + Data **전층 중복** |

### 2.2 이미 완료된 항목

| 항목 | 상태 |
|------|------|
| if-else 단위 분기 → `Map` + `convertAll` | ✅ `ConversionEngine`, 파사드 동일 |
| 매직 넘버 3.28084 / 1.09361 | ✅ `ConversionConstants` + 설정 로드 |
| meter 허브 `convert()` 단일화 | ✅ `ConversionEngine.convert` |

### 2.3 BCE 패키지 (이미 존재)

| 레이어 | 주요 클래스 |
|--------|-------------|
| entity | `UnitRegistry`, `ConversionEngine`, `ConversionConstants`, `ConversionResult` |
| boundary | `LineParser`, `InputValidator`, `PlainOutputRenderer`, `JsonOutputRenderer` |
| data | `ConfigurationLoader`, `UnitsConfigDto` |
| control | *(미분리 — `App.main`이 담당)* |

## 3. 리팩토링 대상 목록 (우선순위 순)

| 순번 | 대상 | 문제 | 적용 기법 | 우선순위 |
|:----:|------|------|-----------|:--------:|
| 1 | `src/main/java/UnitConverter.java` (파사드 전체) | Entity·Boundary·Data·Control이 한 클래스에 중복 구현 (`parse`, `convert`, `renderPlain`, `loadConfig`) | **BCE 위임 + Facade 축소** — 기존 BCE 호출만 남기거나 테스트를 BCE로 이전 후 삭제 | **높음** |
| 2 | `com.unitconverter.App.main()` (13–35행) | Control(오케스트레이션) + Boundary(`printf` 출력) 혼재, `PlainOutputRenderer` 미사용 | **Extract Class** — `ConvertLengthUseCase`(Control) + `ConsolePresenter`(Boundary), `main`은 진입만 | **높음** |
| 3 | `UnitConverter.renderPlain()` / `parseAndConvert()` (62–84행) | 파싱 이중 수행, `parse()` 미경유 시 예외 메시지 불일치 | **Compose Method** + **Replace Duplicate with BCE call** | **높음** |
| 4 | `UnitConverter.loadConfig()` (101–123행) | Data(I/O) + Entity(레지스트리) 혼재, `catch (Exception)` 포괄 삼킴 | **Move Method** → `ConfigurationLoader`, **Replace Error with DomainException** | **중음** |
| 5 | `UnitConverter.parse()` vs `boundary.parser.*` | Boundary 로직·ERR 메시지 중복 | **R-U1** + **R-U2** — 파서·검증 단일화, `ErrorCodes`/`Messages` 상수화 | **중음** |
| 6 | `UnitConverter` 내 `metersPerOneUnit` / `ConversionRow` | `UnitRegistry`·`ConversionResult`와 이중 모델 | **R-L1** — Registry/VO 단일화, 파사드 DTO 제거 | **중음** |
| 7 | `roundOneDecimal` / `App.roundOne` / `PlainOutputRenderer.roundOne` | 반올림 정책(1자리) 3곳 중복 | **Extract Class** — `RoundingPolicy` 또는 `OutputFormatPolicy` | **낮음** |
| 8 | 루트 `UnitConverter.java` | 진입점 이중 개념 | **유지** — `App` 또는 `Application` 한 줄 위임 (현재 양호) | **낮음** |

## 4. 리팩토링 후보 목록

| ID | 내용 | 현재 상태 | Refactoring Wave | 연결 순번 |
|----|------|-----------|------------------|:---------:|
| **R-U1** | 입력 파싱을 `InputParser`로 분리 | `LineParser`·`ParsedInput` **이미 존재** — 파사드 `parse()`만 중복 | Wave 2 | 5 |
| **R-U2** | 예외 메시지·코드 상수화 | ERR 문자열이 `LineParser`/`UnitConverter.parse`에 **중복** | Wave 3 | 5 |
| **R-U3** | 출력 포맷을 `OutputFormatter`로 분리 | `PlainOutputRenderer` **이미 존재** — `App.main`·`renderPlain` 미사용 | Wave 1 | 2, 3 |
| **R-L1** | 변환 비율을 `ConversionRule` VO로 교체 | `UnitRegistry` + `metersPerOneUnit` **이중** | Wave 2 | 6 |
| **R-L2** | if-else → `UnitRegistry` | **완료** | — | — |
| **R-L3** | 3.28084/1.09361 → 상수·설정 | **완료** (`ConversionConstants` + config) | 검증만 | — |
| **R-L4** | `convert()` meter 허브 단일화 | **완료** (`ConversionEngine`) | Wave 2 (위임만) | 1 |

### 4.1 권장 실행 Wave

```text
Wave 0 (선행): 전체 테스트 GREEN + Golden Master PASS
Wave 1: R-U3 + App.main → Control/Boundary 분리 (순번 2)
Wave 2: 파사드 축소·BCE 위임 (순번 1, 3, 6) — R-U1, R-L1, R-L4 정리
Wave 3: loadConfig·예외 상수화 (순번 4, 5) — R-U2
Wave 4: 반올림 정책 통합 (순번 7)
```

## 5. ECB 목표 구조 (Refactoring 완료 시)

```text
루트 UnitConverter.main
    → control.Application / App
          → LineParser + InputValidator     (Boundary, R-U1)
          → ConvertLengthUseCase            (Control)
          → ConversionEngine + UnitRegistry (Entity, R-L1/L4)
          → PlainOutputRenderer             (Boundary, R-U3)
          ← ConfigurationLoader             (Data, R-L3)

파사드 UnitConverter (선택)
    → UseCase thin delegate 또는 테스트 전용 제거
```

**의존 방향:** `Boundary` → `Control` → `Entity` ← `Data`

### 5.1 새 단위 추가 시 변경 최소화

| 시나리오 | 변경 위치 | Java 코드 수정 |
|----------|-----------|----------------|
| 기본 시드에 영구 추가 | `ConversionConstants` + `UnitRegistry.withDefaults()` | 없음 (등록만) |
| 설정 파일로 추가 | `config/units.json` + `ConfigurationLoader` | 없음 |
| 런타임 등록 | `UnitRegistry.register(...)` | `ConversionEngine` 수정 없음 |

**OCP 규칙:** 단위명 `if/switch` 금지 · 비율은 상수/설정만 · `main`/파사드에 로직 추가 금지.

## 6. 테스트 선행 필요 항목

| # | 항목 | 이유 | 현재 상태 |
|---|------|------|-----------|
| T-0 | **전체 `mvn test` GREEN** | Refactoring 게이트 (RG-06) | 35 tests PASS |
| T-1 | **Golden Master PASS** | PLAIN 출력 계약 고정 | `GoldenMasterTest` 4건 + `golden_master_expected.txt` |
| T-2 | **TC-A-01~07, TC-B-01~07** | 파사드 API 계약 | `UnitConverterTest` |
| T-3 | **Dual-Track BCE 테스트** | Control 분리 후 Domain/Boundary 회귀 | `DomainLogicRedTest`, `UiBoundaryRedTest` |
| T-4 | **`App` E2E 스모크 (권장)** | `main` 경로는 Golden/파사드에 미포함 | **없음 → Wave 1 전 추가 권장** |
| T-5 | **ERR-* golden (선택)** | R-U2 상수화 시 BT 스냅샷 | PRD RG-02 |

**테스트 없이 Refactoring을 시작하면 안 되는 이유:**  
구조 변경 후 **동일 입력·동일 출력**을 자동 증명할 수 없어, ECB 분리 과정에서 **조용한 회귀**가 통합 브랜치에 섞일 수 있다.

## 7. 리팩토링 후 검증 방법

### 7.1 실행 명령어

```bash
# 1) 전체 단위·통합 테스트 (필수)
mvn clean test

# 2) Golden Master만 (PLAIN 출력 회귀)
mvn test -Dgroups=golden_master

# 3) 커버리지 (PRD 목표 확인)
mvn clean test jacoco:report
# 리포트: target/site/jacoco/index.html

# 4) 비율 인라인 재발 검사 (수동·리뷰)
# src/main/java 에 3.28084 / 1.09361 리터럴 없음 확인
# 허용: ConversionConstants.java, 테스트 assert

# 5) Golden 기준 갱신 (출력 계약 의도 변경 시만)
.\scripts\generate-golden-master.ps1
git diff src/test/resources/golden_master_expected.txt
```
